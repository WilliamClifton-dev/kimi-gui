# Kimi Code 新手助手

## 一句话定义

面向中文新手的 Kimi Code 本地启动与排障助手：不复制官方聊天界面，帮助用户从“不会安装和配置”走到“成功完成第一次代码任务”。

## 问题定义

目标用户不是缺少另一个 AI 聊天窗口，而是卡在使用 Kimi Code 之前：

- 不知道是否安装成功，也看不懂环境报错
- 不会进入项目目录，不理解 API Key、Base URL、模型和登录方式
- 遇到网络、权限、配置或运行时错误时不知道下一步
- 不知道第一条有效任务该怎么描述，也不理解审批风险

官方 `kimi web` 已覆盖会话、文件引用、审批、Plan、工具输出和模型交互。再次实现这些能力，只会得到维护成本更高的非官方副本。

## 推荐方向

把 Kimi GUI 收缩为 **Kimi Code 新手助手（Beginner Companion）**。产品只负责官方体验前后的缺口，实际编码会话优先启动官方 Kimi Web、CLI 或编辑器集成。

核心承诺：

> 用户不需要先学命令行，也能检查环境、完成配置、选择项目，并启动第一次可用的 Kimi Code 任务。

## MVP 范围

### 1. 环境体检

- 检测操作系统、Kimi Code 安装状态和版本
- 检测登录、配置文件、默认模型和网络可达性
- 用中文解释检测结果，不直接丢出原始错误

### 2. 引导安装与配置

- 根据系统给出单步操作
- 优先使用 Kimi Code OAuth 登录
- 高级用户可配置兼容供应商，但不承诺 Kimi 原生能力完全等价
- 密钥使用系统安全存储，不写入仓库、日志或界面截图

### 3. 项目启动器

- 图形化选择本地项目，无需手动 `cd`
- 启动官方 Kimi Web、CLI 或 VS Code 集成
- 记住最近项目和首选启动方式

### 4. 首次任务向导

- 提供少量真实任务：理解项目、定位错误、补测试、实现小改动、检查变更
- 启动前说明 Kimi 将读取什么、可能执行什么、何时需要批准
- 将任务交给官方运行时执行，不维护第二套会话协议

### 5. 恢复中心

- 识别安装失败、未登录、无效配置、网络失败、权限不足和端口占用
- 每个问题提供明确修复动作和再次检测按钮
- 生成已脱敏的诊断包，便于提交 Issue

## 明确不做

- 不做通用 AI 聊天客户端：与 Kimi、ChatGPT 等官方产品正面重复
- 不重做 `kimi web`：官方功能更多，协议变化也更快
- 不追求 Codex 或 Claude Code 的界面复刻：外观不是产品价值
- 不在 MVP 实现 MCP、插件、子 Agent 管理：先验证新手是否真被安装和配置阻塞
- 不把多供应商作为主卖点：它会稀释 Kimi 新手定位，并增加兼容承诺

## 关键假设与验证

- [ ] 新手主要卡在首次安装、登录、配置和选目录，而非聊天界面。找 5 名目标用户观察首次使用。
- [ ] 助手能把“安装到第一次成功任务”的中位时间降低至少 50%。记录全程时间和失败点。
- [ ] 至少 4/5 用户无需手写终端命令即可完成首次任务。
- [ ] 官方 Kimi Web 暂时不会覆盖完整的桌面环境体检和自动恢复。每个版本发布前复查官方能力。

## 成功标准

- 5 名目标用户中至少 4 名独立完成首次任务
- 安装到首次成功的中位时间不超过 10 分钟
- 正常路径无需用户输入命令
- 常见失败都有中文原因、修复动作和重试入口
- 产品价值可以用“安装、体检、启动、恢复”解释，而不是“换了皮肤的 Kimi”

## 停止条件

满足任一条件就停止独立产品开发，转为教程或向上游贡献：

- 官方产品已提供无终端安装、配置、项目启动和恢复
- 5 名目标用户使用官方流程也能顺利完成首次任务
- 助手无法显著降低首次成功时间或失败率
- 差异最终只剩视觉样式
- 跟随上游变化的维护成本长期高于新手获得的价值

## 当前判断

**有条件继续。** 停止当前完整 GUI 和 Codex 化改造，只做一个可验证的环境体检与项目启动垂直切片。验证通过再扩展；验证失败则及时停止。
